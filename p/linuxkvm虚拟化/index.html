虚拟化基础

术语与示例图

虚拟机

简介

虚拟计算机也称为“虚拟机”（VM，Virtual Machine），它是一种严密隔离且内含操作系统和应用的软件容器。

特征

  • 分区
  • 隔离
  • 封装
  • 独立于硬件

Hypervisor

简介

Hypervisor 是一种运行在基础物理服务器vhe操作系统之间的中间软件层，其可以允许多个 VM 操作系统和 VM 应用共享宿主机底层的内存、CPU、磁盘等物理硬件，也可叫做 VMM（virtual machine monitor），即虚拟机监视器。

注意：其实在术语与示例图中的 vmware workstation 与 kvm 环境 就是 Hypervisor，其实也就是用来管理虚拟机的软件。

系统调用与虚拟场景的系统调用

  1. 系统调用

为了协调多任务，操作系统被分成了两段： 内核空间：接近硬件，主要执行特权指令 用户空间：执行普通的指令 用户空间的执行想要执行某些特权指令，就需要进行"系统调用"

  1. 虚拟场景下的系统调用

虚拟化的两种风格

表现样式

架构差异

  1. Type-I：也成为裸机 Hypervisor。

Type - I 的 Hypervisor 直接运行在硬件之上，没有宿主机操作系统，直接控制硬件资源和客户机。

  1. Type-II：也叫宿主机形 Hypervisor。

Type - II 的 Hypervisor 运行在一个宿主机操作系统之上，作为宿主机操作系统中的一个应用程序，客户机则是在宿主机操作系统上的一个进程。

特点

  1. Type - I：

由于直接运行在硬件上，没有中间的宿主操作系统层，Type - I Hypervisor 具有较低的系统开销和更高的性能效率，能更充分地利用硬件资源，适合对性能要求较高的生产环境。

  1. Type - II：

运行在宿主操作系统之上，虚拟机访问硬件资源需要经过宿主操作系统的内核，会带来一定的性能损耗。在一些对性能要求不特别苛刻的场景，如开发测试环境或个人使用场景中，这种性能损耗通常是可以接受的。

kvm 简介

官方网址

https://www.linux-kvm.org/page/Main_Page

原理介绍

前提

KVM 本身不执行任何硬件模拟，需要使用一个客户空间的程序通过 /dev/kvm 接口设置一个客户机虚拟服务器的地址空间，向它提供模拟的 I/O，并将它的视频现实映射会宿主的显示屏。目前这个应用程序通常使用 QEMU。

实现原理

kvm依赖于HVM技术，只能运行在支持硬件虚拟化的CPU上，也就是说CPU支持VT-x或AMD-v，并且只支持x86_64bits系统。只要基于硬件的操作系统内核一旦装载了kvm模块，那么kvm模块就和linux内核组合成了hypervisor，变成了type-I模型的虚拟机。 kvm类似于寄生虫，linux内核一旦被寄宿，就没有自主权，跟着kvm一起变成hypervisor的一部分。原来的用户空间就变成了hypervisor的管理控制台运行位置，而kvm以字符设备/dev/kvm的形式作为kvm整个的调用管理接口。 kvm创建的虚拟机就是一个进程，ps查看，kill杀死

管理软件

KVM 是运行在单机的系统，比如运行在A宿主机上的虚拟机，是不允许操作B宿主机上的虚拟机。如果实现多机的环境，需要其他软件实现跨主机的统一管理。常见的虚拟化管理平台如下：

参考资料：http://www.linux-kvm.org/page/Management_Tools

  1. oVirt：功能强大，是Redhat虚拟化管理平台RHEV的开源版本。http://www.ovirt.org/

  2. WebVirtMgr：virt-manager的Web模式的替代品,管理单主机的虚拟机。https://www.webvirtmgr.net

  3. OpenStack：最主流的开源虚拟化管理平台,基于Python语言开发实现，适合大型企业。https://www.openstack.org/

  4. Proxmox virtualization environment：简称PVE，是一个开源免费的基于Linux的企业级虚拟化方案，功能不输专业收费的VMware。是一个完整 的企业虚拟化开源平台。借助内置的Web界面，您可以轻松管理VM和容器，软件定义的存储和网络，高可用性集群以及单个解决方案上的多个开箱即用工具。

  5. VMware vSphere：VMware 公司商业化，只用于管理ESXi 的虚拟机，不能管理KVM虚拟机。

发生异常

  1. 关闭 虚拟化服务

  1. 使用管理员身份运行cmd

  1. 执行如下命令
1

参数解析：
--virt-type kvm 指定创建的虚拟机基于的虚拟化技术
--name CentOS-7-x86_64 指定创建虚拟机的名字，推荐"简洁"
--ram 1024 指定创建虚拟机的内存大小
--cdrom=XX.iso 指定创建虚拟机使用的系统镜像
--disk path=XX.raw 指定创建虚拟机使用的虚拟磁盘
--network network=default 指定创建虚拟机使用的网络类型
--graphics vnc,listen=0.0.0.0 指定访问虚拟机使用的客户端配置信息
--noautoconsole 禁用配置的客户端自动连接虚拟机

创建

虚拟机创建的方式除了初始方法，还有一种经常使用的方法–使用配置文件，而使用配置文件的话，会有两种创建方式：

  1. 临时创建
  • 借助于现成的配置文件和之前创建的虚拟机数据，创建一个新虚拟机，他会自动清除
  • 步骤：配置文件-创建虚拟机-查看
  • 命令：virsh create file_name.xml
  1. 正规创建
  • 借助于现成的配置文件在虚拟机管理程序中注册一个新的虚拟机，他不会自动清除
  • 步骤：配置文件-注册虚拟机-开启虚拟机-查看
  • 命令：virsh define file_name.xml
临时创建
  1. 拷贝文件
1
2
3

cd /etc/libvirt/qemu
cp CentOS-7-x86_64.xml CentOS-7-x86_64-1.xml
ls

  1. 修改配置文件

注意：
主要修改的地方： 虚拟机的名字和uuid；
虚拟机磁盘不需要修改，如果要修改的话，一个新建的虚拟磁盘里面因为没有boot文件，就会报错；
网卡什么的尽量不要更改，更改后即使可以，但是启动不成功

  1. 查看两个配置文件的区别，验证只有name和uuid不一样
1

diff CentOS-7-x86_64.xml CentOS-7-x86_64-1.xml

  1. 关闭占用磁盘文件的虚拟机

注意：因为一个运行中的vm是占用硬件磁盘的，所以会导致资源冲突，为了避免资源冲突，我们将另外一个虚拟机关闭

1

virsh shutdown 1

可以看到，临时虚拟机关闭以后，就没有记录了。

临时虚拟机关闭以后自动删除，永久虚拟机就算关闭了还会继续存在。

注意：后面的数字 n 代表虚拟机启动的端口号，特指 VNC 端口号 5900+n

使用 console 登录

默认情况下，新安装的虚拟机，是无法通过 virsh console 命令连入虚拟机中，这时我们需要开启虚拟机的 console 功能。我们可以通过 grubby 来配置 grub 信息。

关机
  • 命令格式
1

virsh shutdown < vmhost_name or vmhost_id >

注意：shutdown 方法依赖于 acpid 服务，Acpid 是一个用户空间的服务进程，它充当 linux 内核与应用程序之间通信的接口，负责将 kernel 中的电源管理事件转发给应用程序。所以要保证我们的操作系统有acpid服务。

  • centos 安装 acpid 服务
1
2
3
4
5

# 对 ubuntu 来说，它会自动安装相关软件并运行该服务
# 对 centos 来说，它需要手动安装并启动
yum install acpid -y
systemctl start acpid
systemctl enable cpid
  1. 宿主机：关闭虚拟机
1

virsh shutdown 3

  1. 宿主机：查看所有虚拟机
1

virsh list --all
