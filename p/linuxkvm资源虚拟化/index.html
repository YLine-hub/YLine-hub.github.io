硬件辅助虚拟化

CPU 虚拟化技术是实现虚拟化的基础，它允许在单个物理 CPU 上同时运行多个虚拟机。英特尔的 VT 和 AMD 的 AMD-V 就是为了支持高效的 CPU 虚拟化而开发的硬件辅助虚拟化技术。

英特尔VT技术

VT-x: 引入了新的操作模式，即 VMX（Virtual Machine eXtensions）模式。

  该模式包含根操作模式（VMX root operation）和非根操作模式（VMX non-root operation）。虚拟机监视器（VMM）运行在根操作模式下，负责管理和调度虚拟机；而虚拟机则运行在非根操作模式下。   当虚拟机执行敏感指令时，会触发 VM 退出（VM exit），将控制权交给 VMM，VMM 处理完后再通过 VM 进入（VM entry）将控制权交还给虚拟机

VT-d

主要针对 I/O 虚拟化，通过直接 I/O 分配技术，允许虚拟机直接访问物理 I/O 设备，减少了 VMM 的干预，提高了 I/O 性能。它通过引入 I/O 内存管理单元（IOMMU），将设备的物理地址映射到虚拟机的虚拟地址空间，实现了设备的隔离和直接访问。

AMD-V技术

SVM（Secure Virtual Machine）

  与英特尔的 VT-x 类似，AMD-V 引入了虚拟机控制结构（VMCB）来管理虚拟机的状态。它也支持两种操作模式：虚拟机模式（VM mode）和主机模式（host mode）。   当虚拟机执行敏感操作时，会发生 VMEXIT 事件，控制权转移到主机模式下的 VMM，VMM 处理后再通过 VMENTRY 事件将控制权返回给虚拟机。

IOMMU

  和英特尔的 VT-d 类似，AMD 也提供了 I/O 虚拟化的支持，通过 IOMMU 实现设备的直接分配和地址转换，提高 I/O 性能和安全性。

实现虚拟机操作宿主机的资源的方式很多，主要是通过以下几种方式来实现：模拟、完全虚拟化、硬件辅助虚拟化、半虚拟化

功能定位

  在虚拟化环境中，虚拟机里的操作系统运行时会执行各种指令，其中部分指令属于特权指令。在传统虚拟化方式下，当虚拟机执行特权指令时，会触发陷入（trap）机制，将控制权转交给宿主机的虚拟机监视器（VMM），VMM 再模拟执行这些指令，最后把控制权交回虚拟机。这种频繁的上下文切换和指令模拟操作会产生较大的性能开销。

  而二进制转换技术是在虚拟机执行特权指令的瞬间，直接把虚拟机的二进制指令转换为宿主机可以直接执行的二进制指令。这样一来，宿主机就能直接执行转换后的指令，无需再经过复杂的陷入 - 模拟 - 返回流程，大大减少了性能损耗。

指令捕获：
    VMM 实时监控虚拟机的指令执行情况，一旦检测到虚拟机要执行特权指令，就立刻捕获该指令。

翻译规则匹配：
    VMM 依据预先设定的翻译规则，把捕获到的虚拟机特权指令映射成宿主机对应的特权指令。这些翻译规则会考虑虚拟机和宿主机的指令集架构、操作系统等方面的差异。

指令转换：
    按照匹配到的翻译规则，把虚拟机的二进制指令转换为宿主机能够直接执行的二进制指令。

执行转换后的指令：
    宿主机直接执行转换后的特权指令，执行完成后把结果反馈给虚拟机。

内存虚拟化

基础知识

内存虚拟化

内存本身就是虚拟化的。因为进程看到的是线性地址空间，内核看到的是物理地址空间。

  没有虚拟化的时候，主机内核以为自己操作的是整个内存空间，而在虚拟化场景中，每个虚拟机内核都认为自己操作的是整个内存空间，这是不现实的。

基本原理

TLB

TLB（Translation Lookaside Buffer）即快表，是一种高速缓存，用于存储最近访问的虚拟地址到物理地址的转换映射，可加速地址转换过程。

  在非虚拟化环境中，操作系统直接管理 TLB，当进程进行地址转换时，会先在 TLB 中查找映射关系，如果找到则直接使用，找不到则进行页表查询并更新 TLB。

  而在虚拟化环境下，存在多个虚拟机，每个虚拟机都有自己的虚拟地址空间和页表，这就需要对 TLB 进行特殊处理，以保证不同虚拟机之间的隔离和高效运行。

虚拟化技术

影子页表

为每个虚拟机维护一份影子页表，影子页表是宿主机物理地址到虚拟机物理地址的映射。当虚拟机进行地址转换时，VMM（虚拟机监视器）会将虚拟机的虚拟地址转换为影子页表中的物理地址，然后再通过宿主机的 TLB 进行查找。

  虚拟机进程经过CPU的MMU的转换到虚拟机的内存中找到了相应的地址，但是这个地址不是真正的宿主机的内存地址，所以再次经过shadow page table的技术，将虚拟的内存地址转换成真正的物理地址，找到对应的存储元素。

缺点：
  shadow page table 必须和虚拟机一一对应使用
  维护影子页表的开销较大: TLB缓存相应的虚拟机A地址转换信息，但是当切换虚拟机B使用的时候，TLB内容就失效了。

mmu虚拟化

MMU（Memory Management Unit，内存管理单元）虚拟化是虚拟化技术里的关键部分，其主要作用是让多个虚拟机能够在同一物理硬件上高效且安全地使用内存资源。

  在非虚拟化环境中，MMU 负责把虚拟地址转换为物理地址，从而让操作系统能够更有效地管理内存。而在虚拟化环境下，由于存在多个虚拟机，每个虚拟机都有自己的虚拟地址空间，这就需要对 MMU 进行虚拟化处理，使得虚拟机的虚拟地址能够正确映射到宿主机的物理地址。

  借助于cpu的硬件虚拟化作用[Intel 的 EPT（Extended Page Tables）和 AMD 的 NPT（Nested Page Tables）]，这些技术允许在硬件层面支持多级页表转换，虚拟机的页表和宿主机的页表可以同时存在于硬件中。

  在虚拟机进行线性地址转换的同时，硬件会自动处理地址转换，通过虚拟MMU将其转换成宿主机内存的物理地址，减少了 VMM 的干预，提升了内存的性能。

缺点：
  TLB的命中率还是比较低

TLB虚拟化

在 MMU（内存管理单元）虚拟化的基础上引入 TLB（Translation Lookaside Buffer，快表）虚拟化功能，在缓存信息的时候，增加一个标签的功能，类似数据库表字段。可以进一步优化虚拟化环境下的地址转换性能。

  类似于影子页表，为每个虚拟机维护一个影子 TLB。当虚拟机进行地址转换时，首先在其影子 TLB 中查找映射信息，如果找到则直接使用，找不到则通过 MMU 的虚拟化机制（如影子页表或硬件辅助的页表）进行查找，并更新影子 TLB。

优势解读

提高地址转换性能：
    通过在 TLB 中缓存常用的地址映射信息，可以减少 MMU 对页表的访问次数，从而加快地址转换速度。

增强虚拟机隔离性：
    TLB 虚拟化可以确保不同虚拟机的地址映射信息不会相互干扰，提高了虚拟机之间的隔离性和安全性。

优化内存使用效率：
    合理管理 TLB 可以减少 TLB 缺失的情况，避免因频繁的页表查找而导致的内存访问延迟，提高内存使用效率。

  主机用户空间中的一个程序，发出一个信息报文，通过系统调用，由内核调用该设备的驱动程序，由主机硬件中的网卡设备将信息报文输出

虚拟化技术

实现虚拟机操作宿主机的资源的方式很多，主要是通过以下几种方式来实现：模拟、半虚拟化、透传、硬件虚拟化

模拟

完全使用软件来模拟真实的常见的硬件使用场景。

  虚拟主机用户空间中的一个程序，发出一个信息报文，通过系统调用，由内核调用该设备的驱动程序，由主机硬件中的网卡设备将信息报文输出，该信息交给宿主机的hypervisor上对应的网卡设备，然后把该网卡接收到的信息暂存到IO栈中，然后调用宿主机的网卡设备驱动，经宿主机网卡将该信息报文发出去。

缺点：
  虚拟机上的设备驱动以及虚拟网卡发送信息所做的工作和hypervisor是重复的。
  物理硬件的性能发挥60%左右

半虚拟化

借助硬件虚拟化功能，将设备驱动分成两部分，前端驱动和后端驱动，前端驱动是虚拟机专用的，后端驱动是统一使用的。

  虚拟主机用户空间中的一个程序，发出一个信息报文，通过系统调用，内核不做任何处理，直接由该设备的虚拟驱动程序–前端驱动，该信息交给宿主机的hypervisor上对应的网卡设备，然后把该网卡接收到的信息暂存到IO栈中，然后调用网卡驱动–后端驱动，经宿主机网卡将该信息报文发出去。
  这样就省略了虚拟机中重复调用驱动发送信息的动作。可以使用物理硬件95%的性能

缺点：
  前后端驱动的方法只适用于硬盘和网卡设备，
  显卡设备的虚拟，一般使用frame buffer机制来实现。目前显卡的虚拟，可以在Centos7上使用半虚拟化方式来实现。

透传

  宿主机上有多块盘，每个虚拟机直接使用一块专用盘，像这种场景方法就叫IO透传。前提是我们的硬件必须支持响应的透传技术.

  宿主机上的hypervisor管理所有的存储设备，然后分配给VM虚拟机一个专用的存储设备，虚拟机使用的时候，直接通过内核空间中的设备驱动，操作该分配的存储设备。

硬件虚拟化

  传统的x86架构的设备有一个集中式的管理设备叫DMA(直接内存访问)，它是一种加速io访问性能的方式，在它内部有一个IOMMU，它可以进行IO寄存器和IO设备端口的自动映射工作。

  如果在宿主机用DMA，所有的虚拟机都连接一个DMA，所以虚拟机io设备的调用必须在IOMMU级别上实现隔离，VT-d就是基于北桥的硬件辅助的虚拟化技术，提高了IO设备的可靠性、灵活性和性能。

网络虚拟化

虚拟原理

虚拟网络设备创建的原理：

  虚拟网络设备是由软件实现的，它们是成对出现的：即虚拟机和hypervisor会生成一对虚拟网卡设备，vm通过vnet0和绑定在hypervisor上的vnet0进行通信，反之一样。

  该虚拟网络设备主要有两种方式：TUN(基本不用)与TAP，TAP模拟以太网设备，操作第二层数据包。TUN模拟网络层设备，操作第三层数据包。

网络模式

网络的虚拟化，一般是由四种模型来实现的：桥接模型、路由模型(仅主机模型)、net模型、隔离模型

桥接模型

桥接模式的核心原理是创建一个虚拟网桥设备，这个网桥就像是一个物理的以太网交换机。虚拟机的虚拟网卡和宿主机的物理网卡都会连接到这个虚拟网桥上。

  当虚拟机发送网络数据包时，数据包会先到达虚拟网桥，然后由网桥根据目标 MAC 地址将数据包转发到合适的端口，可能是宿主机的物理网卡，也可能是其他连接到网桥的虚拟机的虚拟网卡。

路由模型型(仅主机模型)

  宿主机就像一个路由器，它有自己的网络接口连接到外部网络，同时通过虚拟网络接口与虚拟机进行通信。当虚拟机需要与外部网络通信时，数据包会先到达宿主机，然后由宿主机根据路由表将数据包转发到合适的目标网络。

net模型

NAT 模式的核心原理是在宿主机上创建一个虚拟的 NAT 设备，它就像一个网络地址翻译器。

  虚拟机使用私有 IP 地址，处于一个与宿主机隔离的虚拟网络中。当虚拟机要与外部网络通信时，NAT 设备会将虚拟机数据包中的源 IP 地址替换为宿主机的公网 IP 地址，同时记录下这个转换关系，以便外部网络返回数据包时能正确地将其转发给对应的虚拟机。

隔离模型

网卡虚拟化中的隔离模式是一种将虚拟机的网络与外部网络或其他虚拟机网络隔离开来的网络配置方式，以满足特定的安全、隐私或测试需求。

  在隔离模式下，虚拟网卡通过特殊的网络配置，使虚拟机处于一个独立的网络空间中。它通常会创建一个专用的虚拟网络，该网络与宿主机的物理网络以及其他虚拟机所在的网络相互隔离。实现方式可以是通过虚拟交换机的设置，将属于隔离模式的虚拟机连接到特定的虚拟端口组，这些端口组之间在网络层是相互隔离的，数据包无法直接在不同隔离网络之间传输。

  虚拟机在隔离模式下，只能与同一隔离网络中的其他虚拟机或设备进行通信。与外部网络的通信被完全阻断，除非通过特定的网络策略或代理机制进行设置。这意味着虚拟机无法直接访问互联网或其他非隔离网络中的资源，外部网络也无法直接访问隔离模式下的虚拟机，从而实现了网络层面的数据隔离。
