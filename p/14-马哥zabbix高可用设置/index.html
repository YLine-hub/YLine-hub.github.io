Zabbix高可用

功能定位

高可用需求

在前面的部署架构中,Zabbix Server 和MySQL都有只有一台主机,存在单点问题，可以使用Zabbix Server的高可用。

基础架构为LAMP或LNMP环境，采用keepalived实现zabbix服务器高可用，保证主服务器或者Httpd宕掉后能切换到从服务器。MySQL数据库做主主同步，保证两边服务器数据的一致性，实现数据库的高可用。

从Zabbix 6.0版本开始，官方提供了高可用解决方案

1
2
3

https://www.zabbix.com/documentation/7.0/zh/manual/concepts/server/ha
https://www.zabbix.com/documentation/6.0/zh/manual/concepts/server/ha
https://www.zabbix.com/documentation/6.0/zh/manual/concepts/server/ha#implementation-details

高可用方案

Zabbix的高可用方案

  通常高可用性(HA)需要在几乎不需要停机的关键基础设施中使用。为了在服务出现任何失败故障时进行故障转移，进行接管。

  Zabbix提供了一个本地的高可用性解决方案，方便设置，不需要任何HA专业知识也可以完成。本地Zabbix HA对于防止Zabbix server的软件/硬件故障或减少维护停机时间是有用的。

  Zabbix高可用模式下，多台Zabbix server作为集群中的节点运行。当集群中的一个Zabbix server处于active时，其他服务器处于standy，随时准备在必要时接管。

高可用方案细节

在server配置中需要两个参数来启动Zabbix server作为集群节点:

1
2
3

HANodeName 集群中每个zabbix server节点名称必须要有且唯一。
    这是将在agent和proxy配置中引用server的名称(例如:zabbix-node-01)。
    如果您没有指定HANodeName，那么服务器将以standalone模式启动。

1
2
3

NodeAddress参数必须为每个节点指定。
    NodeAddress参数(address:port)将被Zabbix前端用来连接到主server节点。
    NodeAddress必须匹配相应Zabbix server的IP或FQDN名称。

在对配置文件进行更改后，重新启动所有Zabbix server。-它们现在将作为集群节点启动。

1
2

server的新状态可以在报表→系统信息查看，也可以用以下命令查看:
    zabbix_server -R ha_status

其他信息

Zabbix HA解决方案由多个zabbix_server实例或节点组成。每个节点独立配置，但数据库是使用同一个实例。

1
2
3
4
5

Zabbix HA节点包括以下几种状态:
    active(活动)
    standby(备用)
    unavailable(不可用)
    stopped(停止)

在同一个Zabbix HA集群中，只有一个实例或节点处于active(活动)状态，standby(备用)节点不进行数据收集、处理或其他任务，并且不监听端口，并保持一个最少的数据库连接。

global_defs {
    router_id ubuntu24-1
    script_user root
    enable_script_security
}

vrrp_script check_haproxy {
    script "/usr/bin/killall -0 haproxy"
    interval 2
    fall 2
    rise 2
}

vrrp_instance web_vip {
    state MASTER
    interface ens33                 # 本机网卡名称
    virtual_router_id 51
    priority 100
    nopreempt
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass secret123
    }
    virtual_ipaddress {
        192.168.172.100
    }
    track_script {
        check_haproxy
    }
}

vrrp_instance db_vip {
    state BACKUP
    interface ens33
    virtual_router_id 52
    priority 90
    nopreempt
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass secret456
    }
    virtual_ipaddress {
        192.168.172.200
    }
    track_script {
        check_haproxy
    }
}

Zabbbix优化思路

通用的优化思路

数据库层面

服务器性能层面

合理配置硬件资源

根据监控规模和业务需求，为 Zabbix Server和Proxy分配足够的硬件资源，包括CPU、内存、磁盘和网络带宽。对于大规模监控，可考虑使用多核CPU、大容量内存和高速存储设备(如SSD)，以满足数据处理、存储和传输的需求。

优化Zabbix进程参数

调整 Zabbix Server和 Agent 配置文件中的进程参数，如StartPollers、StartTrappers、StartDiscoverers等，以适应不同的监控场景和负载情况。增加这些参数的值可以提高数据采集、接收和发现的并发能力，但要根据服务器硬件资源进行合理调整，避免资源过度占用。

监控服务器资源使用情况

使用系统自带的监控工具或第三方监控工具，实时监控Zabbix Server和Proxy所在服务器的CPU使用率、内存使用率、磁盘I/O、网络流量等资源指标。根据监控数据及时发现性能瓶颈，并进行相应的调整和优化。

采用分布式架构

监控项层面

精简监控项

对监控项进行全面梳理，去除不必要的监控指标，只保留对系统运行和业务关键的监控项。避免监控过多无关紧要的信息，以减少数据采集和存储量，降低系统负载。

设置合理的监控频率

根据监控项的重要性和变化频率，设置不同的监控频率。对于关键的性能指标，如CPU使用率、内存使用率、网络带宽等，可设置较高的监控频率(如每分钟采集一次);对于相对稳定的配置信息或变化缓慢的指标，可适当降低监控频率(如每小时或每天采集一次)。

设置合理的监控频率

根据监控项的重要性和变化频率，设置不同的监控频率。对于关键的性能指标，如CPU使用率、内存使用率、网络带宽等，可设置较高的监控频率(如每分钟采集一次);对于相对稳定的配置信息或变化缓慢的指标，可适当降低监控频率(如每小时或每天采集一次)。

使用监控模板和宏

创建和使用监控模板来统一管理同类设备或服务的监控项配置，提高配置效率和一致性。同时，利用宏来动态设置监控项的参数，以便在不同的环境中灵活应用监控模板。

监控项分组和聚合

将相关的监控项进行分组，方便管理和查看。对于一些具有相关性的监控项，可以进行聚合计算，如计算平均值、总和、最大值等，以更直观地反映系统的整体状态。

前端界面层面

启用缓存机制

在 Zabbix前端服务器上启用缓存，如使用Memcached 或 Redis等缓存系统，缓存常用的监控数据、配置信息和页面元素等。通过缓存可以减少对数据库的查询次数，提高前端页面的加载速度和响应性能。

优化前端页面设计

对 Zabbix前端界面进行优化，简化页面布局，减少不必要的图表和信息展示，提高页面的可读性和易用性。同时，合理设置图表的显示参数，如时间范围、数据粒度等，以快速呈现关键信息。

使用异步加载和分页技术

对于大量数据的展示页面，采用异步加载和分页技术，避免一次性加载过多数据导致页面加载缓慢。用户可以通过分页浏览数据，同时后台异步加载下一页的数据，提高用户体验。

监控前端性能指标

使用前端性能监控工具，如 Chrome 开发者工具、GTmetrix等，监控 Zabbix 前端页面的性能指标，如页面加载时间、资源加载时间、HTTP请求数等。根据监控结果找出性能瓶颈，并进行相应的优化，如压缩图片、合并CSS和JavaScribt文件、减少HTTP请求等。

哪些监控项是比较重要

服务器硬件

CPU 使用率:
    反映CPU的工作负载，过高的使用率可能表示系统负载过重，影响系统性能。
内存 使用率：
    显示内存的使用情况，内存不足可能导致系统频繁交换，降低系统运行速度。
磁盘 I/O:
    包括磁盘读写速度、I/O 等待时间等，可帮助发现磁盘性能瓶颈，如磁盘读写频繁可能导致系统响应缓慢。
硬件温度:
    监控服务器硬件设备(如CPU、硬盘等)的温度，过高的温度可能会影响设备的稳定性和寿命，甚至导致硬件故障。

应用程序

应用程序响应时间:
    指从用户发出请求到收到响应的时间，是衡量应用程序性能的重要指标，过长的响应时间会影响用户体验。
应用程序错误率:
    监控应用程序中出现的错误数量和比例，及时发现应用程序的故障和异常，以便快速定位和解决问题。
业务关键指标:
    根据具体业务需求，监控与业务相关的关键指标，如电商网站的订单量、支付成功率，视频网站的播放量、用户活跃度等，这些指标直接反映了业务的运行状况。

小规模实施

【100台】核心目标：快速部署与单点可靠性

1
2
3
4
5
6
7
8

架构:
    单 Zabbix Server + Agent 直连，无需 Proxy，简化网络层级。
配置:
    手动配置为主，模板复用率高(如统一应用LinuX模板)，关注主机级基础指标(CPU/内存/磁盘IO)
管理:
    人工巡检为主，告警阈值宽松(如内存使用率>80%触发通知)，侧重单主机故障响应(如服务进程异常)。
优化:
    数据库无需分表，定期清理历史数据(保留30天)，前端界面聚焦主机列表与最新告警。

中规模实施

【500台】核心目标：分组管理与性能基线

架构:
    引入 Zabbix Proxy (按机房/业务分组)，分散Server 负载，优化跨机房数据采集。
配置:
    模板分层(基础模板+业务模板)，监控项精细化(如区分Web服务器与数据库服务器的指标)。
管理:
    建立性能基线(如CPU平均负载24小时曲线)，告警分级(关键业务高优先级，非关键低优先级)。
优化:
    数据库开启压缩(History表压缩)，使用Zabbix 自带的自动发现(LLD)减少手动配置，网络关注跨VLAN流量。
