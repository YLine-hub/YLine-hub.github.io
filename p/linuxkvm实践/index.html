存储管理

存储池

存储池环境

基础知识

需求

  为了使虚拟机获得更强大的后端存储能力，libvirt 提供了对各种各样的存储介质支持，包括本地文件系统，网络文件系统，iSCSI，LVM 等多种后端存储系统。在虚拟机中主要以存储池(pool)与存储卷(volume)的形式存在。
  libvirt 中的存储管理独立于虚拟机管理。也就是存储池和存储卷的操作独立于虚拟机的操作存在，即使是没有虚拟机的场景。工作中我们可以当虚拟机需要存储资源时再进行灵活分配。

存储池

  KVM平台以存储池的形式对存储进行统一管理。它可以理解为一种映射关系，即将宿主机器上的存储空间形成可被KVM使用的逻辑存储池，以方便虚拟主机存储资源的统一管理。
  宿主机的存储空间，可以是本地的、网络的等kvm支持的任意形式。

基于分区创建存储池：
virsh pool-define-as <存储池名> fs --source-dev <分区名称> --target <挂载目录>

基于磁盘创建存储池：

virsh pool-define-as <存储池名> logical --source-name <vg名称> --target <挂载目录>

virsh pool-define-as <存储池名> iscsi --source-host <存储主机ip> --source-dev <存储目标> --target <挂载目录>

基于nfs创建存储池：
virsh pool-define-as <存储池名> netfs --source-host <nfs主机ip> --source-path <nfs共享目录> --target <挂载目录>

存储卷

存储卷环境

基础知识

需求

有了存储池，代表我们有存储数据的基本前提了，但是还不能正常使用，需要借助于存储卷才可以。

存储卷是一种可以分配给虚拟机使用的存储设备。在虚拟机中与一个挂载点对应，而物理上可以是一个 虚拟机磁盘文件或一个真实的磁盘分区。它一般存放在一个专用的存储池中来使用。

准备工作

存储卷是结合存储池来使用的，所以我们就先来做存储池卷的准备工作：创建存储池

virsh pool-start sswang_pool

克隆、删除

基础知识

需求

当我们定制好应用数据卷之后，可能需要快速的复制一个，这就用到了数据卷的克隆操作；当我们数据卷的使命结束后，我们就要删除数据卷了。

    virsh vol-delete <已存在的存储卷> <存储池>

网络管理

网络基础

基础知识

网络模型

  kvm类型的虚拟机，默认情况下使用的是nat网络，同一网络模型下创建的多虚拟机之间可以彼此通信，这种网络模型与外界网络进行通信依赖于nat服务，nat服务正常的话，该模型的网络是通过数据包的转换来达到通信的效果的，在网络流量比较大的场景下，该模式的网络模型会成为虚拟机的瓶颈。
  生产中我们一般采用桥接的网络模型，这种模型是借助于宿主机的网卡设备，直接与外网进行通信，效果很好。

vim bridge_setup_function.sh
#!/bin/bash

# 定义网桥名称
BRIDGE_NAME="br0"
# 定义网络接口名称
INTERFACE_NAME="ens33"
# 定义网桥的 IP 地址
BRIDGE_IP="192.168.172.12/24"
# 定义默认网关
DEFAULT_GATEWAY="192.168.172.2"

# 检查命令是否可用
check_command() {
    local cmd=$1
    if ! command -v "$cmd" &> /dev/null
    then
        echo "$cmd 命令未找到，请安装相关包。"
        exit 1
    fi
}

# 创建网桥
create_bridge() {
    if brctl addbr "$BRIDGE_NAME" 2>/dev/null; then
        echo "成功创建网桥 $BRIDGE_NAME"
    else
        echo "创建网桥 $BRIDGE_NAME 失败"
        cleanup
        exit 1
    fi
}

# 开启网桥的生成树协议
enable_stp() {
    if brctl stp "$BRIDGE_NAME" on 2>/dev/null; then
        echo "成功开启 $BRIDGE_NAME 的生成树协议"
    else
        echo "开启 $BRIDGE_NAME 的生成树协议失败"
        cleanup
        exit 1
    fi
}

# 关闭网络接口的 IP 地址并启动接口
prepare_interface() {
    if ifconfig "$INTERFACE_NAME" 0 up 2>/dev/null; then
        echo "成功关闭 $INTERFACE_NAME 的 IP 地址并启动接口"
    else
        echo "关闭 $INTERFACE_NAME 的 IP 地址并启动接口失败"
        cleanup
        exit 1
    fi
}

# 将网络接口添加到网桥
add_interface_to_bridge() {
    if brctl addif "$BRIDGE_NAME" "$INTERFACE_NAME" 2>/dev/null; then
        echo "成功将 $INTERFACE_NAME 添加到 $BRIDGE_NAME"
    else
        echo "将 $INTERFACE_NAME 添加到 $BRIDGE_NAME 失败"
        cleanup
        exit 1
    fi
}

# 为网桥设置 IP 地址并启动网桥
configure_bridge_ip() {
    if ifconfig "$BRIDGE_NAME" "$BRIDGE_IP" up 2>/dev/null; then
        echo "成功为 $BRIDGE_NAME 设置 IP 地址 $BRIDGE_IP 并启动网桥"
    else
        echo "为 $BRIDGE_NAME 设置 IP 地址 $BRIDGE_IP 并启动网桥失败"
        cleanup
        exit 1
    fi
}

# 添加默认网关
add_default_gateway() {
    if route add default gw "$DEFAULT_GATEWAY" 2>/dev/null; then
        echo "成功添加默认网关 $DEFAULT_GATEWAY"
    else
        echo "添加默认网关 $DEFAULT_GATEWAY 失败"
        cleanup
        exit 1
    fi
}

# 清理操作
cleanup() {
    brctl delbr "$BRIDGE_NAME" 2>/dev/null
}
# 主函数
main() {
    check_command "brctl"
    create_bridge
    enable_stp
    prepare_interface
    add_interface_to_bridge
    configure_bridge_ip
    add_default_gateway
    echo "网桥配置完成"
}

# 执行主函数
main

镜像管理

磁盘基础

磁盘模式

根据存储数据方式的不同可以分为两种格式，一种是稀疏模式、一种是全镜像模式，全镜像模式无法做快照，IO层面是有qemu模拟的，CPU和内存是有KVM实现的

raw：指定多大就创建多大，直接占用指定大小的空间：老牌的格式了，性能上来说的话还是不错的。由于原生的裸格式，不支持snapshot。在虚拟机迁移方面还是有很大的限制。

qcow2：是openstack默认也是比较推荐的格式，将差异保存在一个文件，文件比较小而且做快照也比较小，空间是动态增长的。现在比较主流的一种虚拟化镜像格式，支持快照功能。

克隆

基础知识

需求

  在某些特殊场景下，我们需要批量安装多台虚拟机，目前我们学习的虚拟机安装的方式是逐一的，效率太低了，有什么方法能解决这个痛点呢？ – 虚拟机克隆

实现方式

1
2
3

kvm虚拟机的克隆分为两种情况。
    (1) KVM主机本机虚拟机直接克隆。
    (2) 通过复制配置文件与磁盘文件的虚拟机复制克隆(适用于异机的静态迁移)。

镜像克隆

1
2

virt-clone -o <被克隆的虚拟机名称> --auto-clone
